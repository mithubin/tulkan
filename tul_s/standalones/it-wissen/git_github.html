Git & GitHub

Wie Versionskontrolle hier tatsächlich funktioniert — drei Zonen, zwei Remotes, ein Deployment-Hook.

Inhalt 1. Wozu das alles 2. Die drei Zonen 3. Kernbefehle 4. Lokal vs. GitHub vs. NUC 5. Workflow hier 6. post-receive-Hook 7. Häufige Fallen

1. Wozu das alles

Git ist kein Backup-System — obwohl es wie eines wirkt. Es ist eine Datenbank von Absichten: was geändert wurde, warum, in welcher Reihenfolge. Das Wichtigste daran ist nicht die Wiederherstellung, sondern dass mehrere Zustände eines Projekts gleichzeitig existieren können (Branches), und dass die Geschichte nicht verloren geht.

Für dieses Setup kommt dazu: Git ist der Deploy-Mechanismus. Ein git push nuc master löst vollautomatisch einen Container-Rebuild auf dem NUC aus. Keine SSH-Session, kein manuelles Kopieren.

Grundregel: Änderungen an tools_nuc/ werden nie manuell auf den NUC kopiert — immer über git push nuc master. Manuelle Kopien ohne Push führen dazu, dass der nächste Push die Änderungen wieder überschreibt.

2. Die drei Zonen

Jede Datei in einem Git-Projekt lebt in einer von drei Zonen. Der größte Verständnissprung passiert, wenn man diese Zonen aufgehört hat zu verwechseln.

Working Tree

Die Dateien wie du sie siehst — dein Editor, der Dateibrowser. Hier passieren alle Änderungen.

ls / vim / gedit

Staging (Index)

Eine Vorschau des nächsten Commits. Nur was hier drin ist, kommt in den Commit.

git add datei.py

Repository

Die unveränderliche Geschichte. Jeder Commit ist ein fester Punkt mit Hash.

.git/ intern
Working Tree → git add → Staging → git commit → Repository

Umgekehrt — um zu sehen was sich wo befindet:

git status          # zeigt: unstaged (rot) + staged (grün)
git diff            # Working Tree vs. Staging
git diff --staged   # Staging vs. letztem Commit
git log --oneline   # alle Commits, kurz
💡 git add -A oder git add . nimmt alle geänderten/neuen Dateien — aufpassen, keine Secrets (.env, Passwörter) dabei. Lieber einzeln: git add datei.py template.html.

Was ein Commit wirklich ist

Ein Commit ist kein "gespeicherter Zustand" — es ist ein Diff plus Metadaten plus ein Zeiger auf den Vorgänger. Git speichert also nicht N Kopien, sondern eine Kette von Änderungen. Jeder Commit hat einen Hash (SHA-1, 40 Zeichen), der eindeutig aus seinem Inhalt berechnet wird.

commit eaee155a3b
Author: Milan
Date:   2026-06-20

    CLAUDE.md: Panel-Header-Standard dokumentiert

Der Hash eaee155 (gekürzte Form reicht meistens) ist der "Name" dieses Commits. Mit ihm kann man immer auf diesen exakten Zustand zurückspringen: git checkout eaee155.

4. Lokal vs. GitHub vs. NUC

Ein lokales Repository kann beliebig viele Remotes kennen — einfach benannte Adressen für andere Repos. Hier gibt es zwei:

origin → GitHub

  • Klassisches Remote auf github.com
  • Dient als Backup / öffentliche Spiegelung
  • Pull Requests, Issues, Code-Review laufen hier
  • kein Deploy-Hook — Push nur archiviert Code
  • git push origin master

nuc → NUC-Server

  • Bare-Repository auf dem NUC: ~/tul.git
  • Hat einen post-receive-Hook
  • Push löst sofort vollständigen Deploy aus
  • Kein Web-Interface, nur Git-Protokoll (SSH)
  • git push nuc master

Ein Bare-Repository hat keinen Working Tree — nur die .git/-Inhalte, direkt im Verzeichnis. Es dient als Empfänger für Pushes. Man arbeitet nicht darin, man schickt Commits hinein.

# Auf dem NUC: ~/tul.git/
# Enthält: HEAD, config, objects/, refs/ — aber keine Arbeitsdateien
git init --bare ~/tul.git   # so wurde es angelegt

Das eigentliche Checkout (Working Tree) liegt dann woanders — in ~/tul/tools_nuc/. Der post-receive-Hook macht dieses Checkout nach jedem Push.

Ein GitHub-Repository ist im Kern auch ein Bare-Repository, gehostet auf github.com-Servern. Dazu kommen Web-Interface, Issues, Actions, Pull Requests. Git selbst "kennt" das alles nicht — das ist alles GitHub-Schicht obendrauf.

# Verbindung prüfen:
git remote -v
origin  git@github.com:milan/mkan.git (fetch)
origin  git@github.com:milan/mkan.git (push)
nuc     user@yourserver:tul.git (fetch)
nuc     user@yourserver:tul.git (push)
# Remote hinzufügen
git remote add nuc user@yourserver:tul.git

# Ersten Push (Branch-Tracking setzen)
git push -u nuc master

# Danach reicht:
git push nuc master

# Remote umbenennen oder entfernen
git remote rename old new
git remote remove nuc

5. Der Workflow hier

Für tools_nuc/ (tul.yourdomain.example) ist der komplette Ablauf:

# 1. Dateien bearbeiten (in tools_nuc/)
# 2. Was hat sich geändert?
git status
git diff

# 3. Relevante Dateien stagen
git add tools_nuc/nach/panel.html tools_nuc/static/tul-files.js

# 4. Commit
git commit -m "nach: Umbenennen-Feature + DV-Refresh"

# 5. Deploy — startet vollautomatisch
git push nuc master

# Optional: auch auf GitHub sichern
git push origin master
Nach dem Push: Der NUC-Hook checkt den Code aus, baut alle geänderten Container neu (--build) und startet sie. In ca. 30–60 Sekunden ist die Änderung live auf tul.yourdomain.example.

Branches hier

Für tools_nuc wird direkt auf master gearbeitet — kein Branch-Overhead für ein Einzelnutzer-Setup. Bei mkan (GitHub-Projekt mit mehr Komplexität) gibt es Feature-Branches und Pull Requests.

# Feature-Branch (für experimentelle Sachen)
git checkout -b feature/neues-modul
# ... arbeiten ...
git checkout master
git merge feature/neues-modul
git branch -d feature/neues-modul

6. Der post-receive-Hook

Auf dem NUC liegt in ~/tul.git/hooks/post-receive ein Shell-Skript, das nach jedem empfangenen Push automatisch ausgeführt wird. Das ist der Kern des Deploy-Mechanismus.

#!/bin/bash
# ~/tul.git/hooks/post-receive

# Commits in den Work Tree auschecken
GIT_WORK_TREE=~/tul/tools_nuc git checkout -f

# Deploy-Skript ausführen
cd ~/tul && bash deploy-tul.sh

deploy-tul.sh macht dann:

1.nginx-Config zusammenbauen: nginx.conf.base + tools_nuc/nginx_tul.conf + schließende Klammer → nexcloc/nginx.conf
2.nginx-Container neu laden (nginx -s reload)
3.docker compose up -d --build — baut nur geänderte Container neu, startet alle
nginx.conf.base-Mechanismus: Beim ersten Lauf von deploy-tul.sh wird alles, was in nexcloc/nginx.conf über dem Marker # ── tul.yourdomain.example steht, als nginx.conf.base gespeichert. Danach wird bei jedem Deploy aus base + nginx_tul.conf + "}" eine neue nginx.conf gebaut. So koexistieren tul mit anderen nginx-Diensten auf dem NUC.

7. Häufige Fallen

✗ Dateien per SCP deployed, dann git push gemacht: Der Hook checkt den Repo-Stand aus — überschreibt dabei alles im Work Tree, also auch die SCP-Änderungen die nicht im Repo waren. Fix: immer erst committen, dann pushen.
✗ git add -A ohne hinzuschauen: Nimmt auch .env, temporäre Debug-Dateien, binäre Daten. Dann sind Secrets im Repository — und git vergisst nichts. Fix: erst git status, dann selektiv adden.
✗ git commit --amend nach Push: Verändert einen bereits gepushten Commit — die Hashes divergieren, ein Force-Push wird nötig. Besser: Fehler im nächsten Commit korrigieren.
⚠ Merge-Konflikte: Entstehen wenn zwei Commits dieselbe Stelle einer Datei unterschiedlich ändern. Git markiert die Konfliktstellen in der Datei mit <<<<<<< / ======= / >>>>>>>. Manuell auflösen, dann git add + git commit.
💡 .gitignore: Dateien die nie ins Repo sollen (Caches, Secrets, Build-Artefakte) in .gitignore eintragen. Wird von git add -A respektiert. Bereits getrackte Dateien werden davon nicht berührt — zuerst git rm --cached datei.
💡 Rückgängig machen: Letzte Änderungen (nicht-staged): git restore datei.py. Staged zurückziehen: git restore --staged datei.py. Letzten Commit rückgängig (lokal, nicht gepusht): git reset HEAD~1. Nie git reset --hard ohne vorherige Überlegung.
