Docker — Grundlagen & tul-Stack

Was ein Image von einem Container unterscheidet, wie Compose alles zusammenhält — und warum Container-Namen DNS-Einträge sind.

Inhalt 1. Wozu Docker 2. Image vs. Container 3. Dockerfile 4. docker-compose 5. Volumes & Bind-Mounts 6. Netzwerk 7. Der tul-Stack 8. Alltagsbefehle

1. Wozu Docker

Das klassische Problem: Software läuft auf dem Entwicklungsrechner, aber nicht auf dem Server. Unterschiedliche Python-Versionen, fehlende Libraries, andere Systemkonfiguration. Docker packt das Programm zusammen mit allem, was es braucht, in ein unveränderliches Paket — das überall gleich läuft.

Für tul.yourdomain.example kommt dazu: acht verschiedene Module (Flask, Python, Shell) laufen isoliert, können unabhängig neugebaut werden, und teilen nur was sie explizit teilen sollen (Volumes, Netzwerke). Kein Dependency-Chaos zwischen trskr und lern.

2. Image vs. Container

Image (unveränderlich)

  • Schreibgeschützter Snapshot: OS-Basis + Dependencies + Code
  • Wird durch docker build oder docker pull erzeugt
  • Hat einen Namen und Tag: tul-hub:latest
  • Wie eine Vorlage — wird selbst nie ausgeführt
  • Kann aus einem Image viele Container erzeugen

Container (laufend)

  • Image + schreibbare Oberschicht + Prozess(e)
  • Hat einen Namen/ID, Ports, eigenes Netzwerk-Interface
  • Änderungen in der schreibbaren Schicht gehen beim rm verloren
  • Persistenz nur über Volumes (außerhalb des Containers)
  • Start: docker run oder docker compose up

Die Analogie: Image = Kuchenrezept, Container = gebackener Kuchen. Aus dem Rezept kann man beliebig viele Kuchen backen. Das Rezept ändert sich nicht wenn du den Kuchen isst.

💡 Ein Container-Neustart (docker restart) behält die schreibbare Schicht. Ein docker rm + Neustart aus dem Image löscht sie. Wer Daten überleben lassen will, braucht Volumes.

3. Dockerfile

Ein Dockerfile ist die Bauanleitung für ein Image — eine Folge von Schichten, die nacheinander ausgeführt werden. Jede Instruktion erzeugt eine Schicht; Docker cached sie. Wenn sich nur die letzten Zeilen ändern, wird nur die betroffene Schicht neu gebaut.

# Beispiel: tul-hub Dockerfile (vereinfacht)
FROM python:3.12-slim

WORKDIR /app

# Dependencies zuerst — damit dieser Layer cached bleibt
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Dann erst den Code — ändert sich öfter
COPY tul_hub/ tul_hub/
COPY tul_auth/ tul_auth/
COPY static/ static/

ENV DOCKER=1
EXPOSE 5000

CMD ["python", "tul_hub/hub_server.py"]
⚠ COPY static/ ist eine bewusste Design-Entscheidung: tul-theme.js, tul-files.js usw. werden in den Hub-Container gebacken. Änderungen an diesen Dateien wirken erst nach einem Hub-Rebuild — der passiert automatisch bei git push nuc master.

Layer-Caching nutzen

Reihenfolge im Dockerfile: was sich selten ändert oben, was sich oft ändert unten. Dependencies (requirements.txt) oben, Code unten. Sonst wird bei jeder Codeänderung alles neu installiert.

4. docker-compose

Compose koordiniert mehrere Container, die zusammen eine Anwendung bilden. Statt acht docker run-Befehle mit vielen Flags zu tippen, steht alles in einer docker-compose.yml.

# docker-compose.yml — Grundstruktur
services:
  tul-hub:
    build: {context: ., dockerfile: Dockerfile.hub}
    container_name: tul-hub
    ports: ["5000:5000"]          # host:container
    volumes:
      - tul_data:/tul_data          # Named Volume
    restart: unless-stopped

  kurv:
    build: {context: ., dockerfile: kurv/Dockerfile}
    container_name: kurv
    ports: ["5008:5008"]
    volumes:
      - tul_data:/tul_data
    restart: unless-stopped

volumes:
  tul_data:                         # Named Volume — überlebt Container-Neustart

Wichtige Compose-Befehle:

BefehlWas er tut
docker compose up -dAlle Services starten (im Hintergrund)
docker compose up -d --buildStarten, vorher Images neu bauen
docker compose downAlle Services stoppen und Container löschen
docker compose restart kurvEinzelnen Service neu starten
docker compose logs -f kurvLogs eines Services verfolgen
docker compose psStatus aller Services
docker compose build hubNur einen Service neu bauen (ohne Start)
docker compose exec kurv bashShell in laufendem Container öffnen

5. Volumes & Bind-Mounts

Container sind flüchtig — was in der Container-Schicht steht, geht beim Löschen verloren. Für persistente Daten gibt es zwei Mechanismen:

Docker verwaltet den Speicherort (/var/lib/docker/volumes/). Der Container sieht einen Pfad, der tatsächliche Ort ist Docker-intern. Überlebt docker compose down, wird erst mit docker volume rm gelöscht.

# In docker-compose.yml:
volumes:
  - tul_data:/tul_data    # Named Volume

volumes:
  tul_data:               # Deklaration am Ende

# Befehle:
docker volume ls
docker volume inspect tul_data
docker volume rm tul_data   # löscht alle Daten!

Im tul-Stack: tul_data enthält /tul_data/db/tul.sqlite (Nutzer, Themes, Dateien) und /tul_data/files/ (hochgeladene Dateien). Wird in alle Module-Container eingebunden.

Ein Pfad auf dem Host wird direkt in den Container eingehängt. Änderungen sind sofort sichtbar, in beide Richtungen. Gut für Entwicklung — für Produktion eher Named Volumes (portabler, weniger Permission-Probleme).

# In docker-compose.yml:
volumes:
  - /home/milnuc/mkan/data:/app/data    # absoluter Host-Pfad
  - ./config:/app/config               # relativer Pfad (zur Compose-Datei)

Bei mkan auf dem NUC: /mnt/tul/db/ ist ein Bind-Mount — so kann direkt mit sqlite3 /mnt/tul/db/tul.sqlite debuggt werden ohne in Container-Shell zu gehen.

6. Netzwerk

Container im gleichen Compose-Projekt sind automatisch im selben Docker-Netzwerk. Das Wichtigste: Container-Namen funktionieren als DNS-Hostnamen.

# kurv-Container kann tul-hub erreichen:
curl http://tul-hub:5000/theme

# nginx-Container (der außerhalb von Compose läuft) auch:
# weil er im gleichen Docker-Netzwerk ist (docker network connect)
💡 Das ist der Grund, warum in nginx_tul.conf steht set $upstream kurv:5008; — kurv ist der Container-Name, und Docker löst ihn intern zu der richtigen IP auf. Kein Eintrag in /etc/hosts nötig.
⚠ nginx proxy_pass mit Variable + Trailing-Slash-Falle: proxy_pass http://$upstream/; (Variable + Trailing Slash) lässt nginx alle Requests als GET / ans Backend schicken — der Pfad geht verloren. Korrektes Muster: rewrite ^/tool(.*) $1 break; + proxy_pass http://$upstream; (kein Trailing Slash).

Port-Mapping

ports: ["5008:5008"] macht den Container-Port 5008 auf dem Host erreichbar. Für Container, die nur intern (über nginx) zugänglich sein sollen, braucht man kein Port-Mapping — der Container ist trotzdem über seinen Namen im Docker-Netzwerk erreichbar.

7. Der tul-Stack

Alle Module laufen in einem einzigen Compose-Stack auf dem NUC. nginx sitzt davor und routet nach Pfad-Prefix.

nginx (nexcloc — separater Stack, externer Container)
tul.yourdomain.example → routet nach Pfad → jeweiliger Container
tul-hub:5000 trskr:5004 bild:5005 popt:5006 lern:5007 kurv:5008 nach:5009 kal-trel:5010 buch:5011
Geteilte Volumes:
tul_data:/tul_data/db/tul.sqlite tul_data:/tul_data/files/

tul-hub ist der Sonderfall: er hat keine eigene Subpfad-Route, sondern bekommt den Catch-all location /. Er liefert die statischen Assets (tul-theme.js, tul-files.js) für alle anderen Module.

Rebuild-Logik: docker compose up -d --build baut nur Images neu, deren Dockerfile oder Kontext sich geändert hat. Alle anderen Container bleiben unangetastet. Ein git push nuc master triggert immer --build, damit Codeänderungen wirksam werden.

Debugging-Befehle für den tul-Stack

# Alle Container-Status
docker compose ps

# Logs eines Moduls live verfolgen
docker compose logs -f kurv

# In einen Container schauen
docker compose exec tul-hub bash

# SQLite direkt abfragen (über Bind-Mount)
sqlite3 /mnt/tul/db/tul.sqlite ".tables"

# Image-Größen
docker images | grep tul

# Volume-Inhalt
docker run --rm -v tul_data:/d alpine ls /d
