1. Server ist eine Rolle, kein Ding

Das klingt banal, ist aber der Schlüssel zum Verständnis fast aller Netzwerkprobleme: Ein Programm ist nicht einfach „ein Server". Es ist ein Server gegenüber diesem Aufrufer, in diesem Moment. Dieselbe uvicorn-Instanz, die dem Browser antwortet, ist gleichzeitig ein Client, wenn sie Anfragen an den OO-Container weiterleitet.

Browser
localhost / mkan.yourdomain.example
Client
⇄
mkan (uvicorn)
:7879
Server Client
⇄
OnlyOffice
Docker :8088
Server Client

Das schafft ein Adressierungsproblem: Wenn OO eine Datei herunterladen oder einen Callback senden will, muss es mkan als Server ansprechen — aber von welcher Adresse aus? Und wie lautet diese Adresse aus OOs Perspektive? Das ist der Kern aller OO-Integrationsprobleme.

Grundregel: Bevor ein Dienst mit einem anderen spricht, muss er wissen: Welche Adresse hat das Gegenüber aus meiner Netzwerkperspektive? Die Antwort unterscheidet sich je nachdem, ob man im gleichen Container, im gleichen Docker-Netz, auf dem gleichen Host oder im Internet spricht.

Lokal (Entwicklung)

✓Kein TLS nötig — Browser und Server auf demselben Rechner
✓uvicorn direkt mit --reload starten, sofortige Codeänderungen
⚠OO-Container braucht host.docker.internal:7879 statt localhost — Docker isoliert sein Netz
⚠Service Worker cached aggressiv — nach HTML-Änderungen manuell löschen
✕OO-IP-Blocking: Docker blockiert standardmäßig private IPs in Callbacks → muss per local.json deaktiviert werden
✕Analytics.js: OO versucht Google Analytics zu laden, scheitert → mkan injiziert Stub in index.html

NUC / Produktion

✓TLS an nginx terminiert — beide Container kommunizieren intern unverschlüsselt (kurzer Weg, sicher)
✓OO und mkan im selben Docker-Netz → OO erreicht mkan über Service-Name mkan
✓Kein IP-Blocking-Problem: Kommunikation bleibt innerhalb des Docker-Netzes
⚠OO_MKAN_BASE muss auf die interne URL zeigen (http://mkan:7879), nicht auf die öffentliche Domain — sonst Umweg über nginx und unnötige TLS-Overhead
⚠Deploy-Prozess: tar + scp + deploy-script — kein hot-reload, Service muss neu gestartet werden
✕OO JWT-Validierung: ab OO 7.x erwartet OO ein JWT in allen API-Requests — entweder JWT_ENABLED=false oder korrektes Secret

2.5 SQLite: eingebettete Datenbank

Eine klassische Datenbank hat einen eigenen Server-Prozess, der läuft — auch wenn keine Anwendung verbunden ist. Die Anwendung schickt SQL als Text über eine Verbindung, der DB-Server führt es aus und antwortet. Zwei Prozesse, eine Prozessgrenze dazwischen.

SQLite dreht das um: Die gesamte Datenbanklogik — SQL-Parser, Transaktionsmanager, Speicherverwaltung — steckt als Bibliothek direkt im Anwendungsprozess. In Python ist das import sqlite3, die Bibliothek ist in die Standardbibliothek einkompiliert. Es gibt keinen DB-Prozess, keinen Daemon, keine Verbindung. Die Bibliothek greift direkt auf die Datei zu.

Die Serverrolle existiert vollständig — SQLite zieht sie in den Prozess hinein, statt sie auszulagern. Der Preis: kein zentraler Koordinator für viele gleichzeitige Schreibzugriffe (deshalb WAL-Mode). Der Gewinn: keine Infrastruktur, kein Setup, die Datenbank ist eine einzige portable Datei.

3. Das OnlyOffice-Dreieck

OO-Integration ist kein normales Client-Server-Verhältnis. Es sind drei Parteien, jede spielt abwechselnd beide Rollen — und jede braucht eine eigene Antwort auf die Frage: „Unter welcher Adresse ist das Gegenüber für mich erreichbar?"

Warum das lokal schwieriger ist als auf dem NUC

Das Docker-Netz-Problem (lokal): OO läuft in einem Docker-Container. Wenn OO localhost:7879 aufruft, meint es seinen eigenen localhost — nicht den des Host-Rechners. Docker-Container haben ihr eigenes virtuelles Netzwerk-Interface.

Lösung: Docker stellt den Hostname host.docker.internal bereit, der auf das Gateway zum Host zeigt. Deshalb: OO_MKAN_BASE=http://host.docker.internal:7879.
Auf dem NUC ist es einfacher: Beide Container (mkan + OO) sind im gleichen Docker-Netz (webproxy). Container können sich direkt über ihren Service-Namen ansprechen: OO ruft http://mkan:7879/... auf. Kein Magic-Hostname nötig.
Das IP-Blocking-Problem (lokal): OO blockiert standardmäßig Requests an private IP-Adressen (10.x, 172.x, 192.168.x) — aus Sicherheitsgründen, damit OO nicht als SSRF-Vehikel missbraucht werden kann. host.docker.internal löst auf eine solche private IP auf → Callback schlägt fehl.

Lösung: local.json in OO mit allowPrivateIPAddress: true überschreiben. Nur lokal nötig.

4. Protokollbesonderheiten

HTTP Einfache Requests

Request → Response, dann fertig. Zustandslos. Jede API-Anfrage trägt JWT im Header. Funktioniert problemlos durch nginx und Docker-Proxies.

✓Standardmäßig überall erlaubt, gut loggbar

WebSocket OO-Kollaboration

Dauerhaft offene bidirektionale Verbindung. OO braucht das für Echtzeit-Mitarbeit (Co-Authoring). mkan tunnelt WS via websockets-Library.

⚠nginx muss WS-Upgrade erlauben (proxy_http_version 1.1, Upgrade-Header)

CORS Same-Origin-Policy

Browser verweigert Requests zu anderen Origins (Schema + Host + Port). OO-JS läuft auf localhost:7879 (via Proxy), darf keine direkten Requests zu localhost:8088 machen.

✓Proxy löst das: alles läuft über /oo/* auf Port 7879
✕Analytics.js: OO lädt von CDN — separater Origin → geblockt → Stub nötig

TLS Verschlüsselung

HTTPS ist an nginx terminiert. mkan und OO sprechen intern HTTP. Kein Problem solange beides im geschlossenen Docker-Netz bleibt.

⚠Lokal: kein TLS. Kein Problem — aber Service Worker auf HTTPS-Sites sind strenger

JWT Zwei Ebenen

mkan nutzt JWT für eigene Auth (sessionStorage). OO hat eigenes JWT-System für seine interne API-Absicherung. Beides getrennt.

⚠OO 7.x+: JWT_ENABLED=false zum Deaktivieren oder eigenes Secret konfigurieren

Service Worker / Cache

Browser-Seite kann sich selbst cachen. Nach Deploy-Änderungen wird die alte index.html aus dem Cache geliefert — OO-Code-Änderungen wirken erst nach SW-Löschung.

✕Lokal besonders schmerzhaft: F12 → Anwendung → SW → Registrierung aufheben

5. Grenzen: technisch vs. konfigurationsbedingt

Nicht alle Probleme haben dieselbe Ursache — und nicht alle lassen sich genauso lösen. Eine technische Grenze ist in der Protokoll-Spezifikation oder Hardware begründet; eine konfigurationsbedingte ist ein gesetzter Default, den man ändern kann.

Faustregel: Wenn ein Problem nur lokal auftritt und auf dem NUC nicht — ist es meistens eine Netzwerkadressierungs- oder Docker-Isolation-Frage (technisch, nicht vermeidbar). Wenn es auf beiden Umgebungen auftritt aber anders konfiguriert sein muss — ist es ein Default-Setting eines Dienstes (konfigurationsbedingt, immer lösbar).
